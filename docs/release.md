# Releases

DimFort uses **tag-driven** releases. Pushing a tag matching `v*.*.*`
triggers [`.github/workflows/release.yml`](../.github/workflows/release.yml),
which runs the test matrix (Linux/macOS/Windows on Python 3.11–3.13),
builds an sdist + wheel, publishes them to PyPI, and creates a
GitHub Release with autogenerated notes. The publish steps are gated
on the test job — a single failing test blocks the release.

## One-time PyPI setup (do this before the first release, not before)

We use **PyPI Trusted Publishing** (OIDC) instead of API tokens.

1. Reserve the project name by registering an account on
   <https://pypi.org/>.
2. Go to
   <https://pypi.org/manage/account/publishing/> and add a *pending
   publisher* with:
   - **PyPI project name**: `dimfort`
   - **Owner**: `ArrialVictor`
   - **Repository name**: `DimFort`
   - **Workflow filename**: `release.yml`
   - **Environment name**: `pypi`
3. In the GitHub repo, create an *environment* named `pypi`
   (Settings → Environments). Optionally add a required reviewer for
   manual approval before every release.

Until both sides are configured, pushing a tag will fail at the
`publish-pypi` step. The sdist/wheel build and the GitHub Release will
still succeed, which is useful for testing the pipeline.

## Cutting a release

1. Bump `version` in [`pyproject.toml`](../pyproject.toml).
2. Add a new section to [`CHANGELOG.md`](../CHANGELOG.md).
3. Commit; PR; merge to `main` once CI is green.
4. **README image-paths preflight.** PyPI renders the README's long-description
   but **does not resolve relative paths** the way GitHub does — relative
   `src="docs/img/..."` / `srcset="docs/img/..."` references render as broken
   images on the project's PyPI page. GitHub renders them fine, so the issue
   only surfaces post-publish. Every image in `README.md` must use the
   absolute `raw.githubusercontent.com` URL (same shape as the social-preview
   line at the top). Quick check, expects no output:
   ```bash
   grep -nE 'src="[^h]|srcset="[^h]' README.md
   ```
   PyPI's README is **frozen** once the version is published — a broken-image
   fix only lands for users on the next release; you can't edit a published
   release's long-description.
5. Tag the merge commit:
   ```bash
   git tag v0.2.0
   git push origin v0.2.0
   ```
6. Watch the `release` workflow on GitHub Actions. The `publish-pypi` job
   pauses for manual approval (the `pypi` environment has required reviewers
   set) — approve once the test + build jobs go green.

## Branch protection (recommended)

Settings → Branches → Add rule for `main`:

- **Require a pull request before merging** (≥ 1 approval).
- **Require branches to be up to date before merging**.
- **Do not allow bypassing the above settings**.
- *(Optional)* **Require signed commits** if you sign your commits.
- *(Optional)* **Restrict who can push** to maintainers only.

`ci.yml` runs the slimmed test + lint job on every push and pull request to
`main` (the full matrix above stays on tag push), so PRs get a status check to
gate merging on. To reproduce it locally:
`pytest -q && ruff check .`.
